# Shipyard CLI And Skill Spec

## Purpose

`shipyard-cli` is a Rust command-line interface for Shipyard. It is used by humans, scripts, and agents through a packaged `SKILL.md`.

The CLI is not a separate agent workflow product. It exposes normal Shipyard product primitives.

## Distribution

Required:

- Written in Rust.
- Built in CI/CD.
- Published as versioned release artifacts.
- Provide binaries for:
  - macOS arm64.
  - macOS x64.
  - Linux x64.
  - Linux arm64 where feasible.
  - Windows x64 if practical.
- Provide checksums.
- Provide install/update command or documented install path.

The `SKILL.md` should fetch a compatible release artifact and verify it before use.

## Configuration

Default config path:

```text
~/.config/shipyard/config.toml
```

Config fields:

```toml
api_url = "https://api.shipyard.example"
default_account = "<owner pubkey>"
output = "human"
```

Secrets:

- Store private keys only if user explicitly chooses local CLI signing.
- Prefer OS keychain where available.
- Support remote signer/NIP-46 where practical.
- Agents should usually use their own pubkey to propose to a human, not hold a human private key.

## Output Modes

All commands support:

- human-readable default output.
- `--json` for automation.

JSON output must be stable enough for `SKILL.md`.

Error behavior:

- Non-zero exit code on failure.
- JSON errors follow API error model when `--json` is set.

## Commands

### Auth

```bash
shipyard auth login
shipyard auth logout
shipyard auth status
```

Login modes:

- local key.
- remote signer.
- auth event generated by external signer if needed.

### Accounts

```bash
shipyard accounts list
shipyard accounts use <owner-pubkey>
```

Shows:

- own account.
- delegated accounts.
- capability flags.

### Delegates

```bash
shipyard delegates invite <delegate-pubkey>
shipyard delegates revoke <delegate-pubkey>
shipyard delegates list
```

Rules:

- Caller must be owner.
- Server-side invite only.

### Proposals

```bash
shipyard propose --to <owner-pubkey> --content <text> --time <iso>
shipyard propose --to <owner-pubkey> --file <path> --queue <queue-id>
shipyard proposals list
shipyard proposals edit <id>
shipyard proposals delete <id>
shipyard proposals reject <id>
shipyard proposals sign <id>
shipyard proposals sign --batch <id>...
```

Agent use case:

- Agent authenticates with its own pubkey.
- Human owner has invited that pubkey.
- Agent proposes an event to human owner.
- Human reviews and signs in web, mobile, or CLI.

### Schedule

```bash
shipyard schedule --content <text> --time <iso>
shipyard schedule --file <path> --time <iso>
shipyard schedule --event-json <path> --time <iso>
shipyard schedule --event-json <path> --queue <queue-id>
shipyard send-now --content <text>
```

Rules:

- Requires signer for active owner.
- If no signer for active owner, command should suggest `shipyard propose --to`.

### Queues

```bash
shipyard queues list
shipyard queues create --name <name> --cadence <duration> --start <iso>
shipyard queues update <id>
shipyard queues archive <id>
shipyard queues next-slot <id>
```

Cadence input supports:

- seconds.
- minutes.
- hours.
- day-like durations.

Output must normalize to seconds in JSON.

### Media

```bash
shipyard media upload <path>
shipyard media servers
```

Rules:

- Uses Blossom only.
- Reads user's kind `10063` list.
- Falls back to `https://blossom.primal.net`.
- Prints uploaded URL.

### Relays

```bash
shipyard relays list
shipyard relays set <relay>...
shipyard relays add <relay>
shipyard relays remove <relay>
```

### Status

```bash
shipyard status
shipyard posts list
shipyard posts show <id>
shipyard posts cancel <id>
shipyard posts retry <id>
```

### DVM

The CLI may help create compatible DVM requests:

```bash
shipyard dvm schedule --dvm <pubkey> --event-json <path> --relay <relay>
```

This must use the unchanged kind `5905` surface.

## `SKILL.md` Package

Purpose:

- Let agents discover, install, configure, and use `shipyard-cli`.

The skill should include:

- What Shipyard does.
- How to install the CLI binary.
- How to verify checksum.
- How to configure `api_url`.
- How to authenticate as an agent pubkey.
- How to propose to a human owner pubkey.
- How to list proposal status.
- Safety rules.

Safety rules for agents:

- Do not ask for or store a human private key.
- Use the agent's own pubkey.
- Propose events to a human owner.
- Do not claim a proposal is published until status confirms owner signed and worker published.
- Use `--json` for machine-readable operations.
- Show the human the content being proposed.

Example skill workflow:

```bash
shipyard auth status --json
shipyard accounts list --json
shipyard propose --to <human-pubkey> --content "..." --time "2026-04-18T10:00:00Z" --json
shipyard proposals list --json
```

## CI/CD Requirements

CI must:

- Run Rust format/lint/test.
- Build release binaries.
- Generate checksums.
- Publish draft or final GitHub release.
- Produce a machine-readable manifest:

```json
{
  "version": "x.y.z",
  "artifacts": [
    {
      "target": "aarch64-apple-darwin",
      "url": "https://...",
      "sha256": "..."
    }
  ]
}
```

The skill should fetch this manifest.

## Acceptance Criteria

The CLI and skill spec is satisfied when:

- A user can install CLI from CI release.
- A user can authenticate and schedule a signed post.
- A delegate can propose to an owner.
- An owner can sign or reject from CLI.
- An agent can use `SKILL.md` to install CLI and propose using its own pubkey.
- All automation-critical commands have stable `--json`.
